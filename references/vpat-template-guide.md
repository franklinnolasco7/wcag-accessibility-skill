# VPAT / ACR Completion Guide

> How to accurately complete a Voluntary Product Accessibility Template (VPAT)
> to produce a legally defensible Accessibility Conformance Report (ACR)

## When to load this file

Load when the user asks how to write, complete, or evaluate a VPAT or ACR;
needs to understand conformance terminology (Supports, Partially Supports, etc.);
is preparing for government or enterprise procurement; or is reviewing a vendor's
accessibility documentation.

Do not load for general WCAG or implementation questions.

---

## What is a VPAT?

A VPAT (Voluntary Product Accessibility Template) is a document created by IT vendors to explain how their products meet accessibility standards. When completed based on real testing, it becomes an **ACR (Accessibility Conformance Report)**.

VPATs are required or strongly preferred by:
- US Federal government agencies (Section 508 procurement)
- Enterprise IT procurement teams
- Educational institutions
- Healthcare organizations subject to Section 504/508

**Key principle:** A VPAT must be based on actual testing, not assumptions or marketing claims. Inaccurate VPATs expose vendors to legal liability and damage customer trust.

## VPAT Versions

| Version | Standards Covered | Use When |
|---|---|---|
| VPAT 2.5 INT (International) | WCAG 2.1, Section 508, EN 301 549 | US + EU customers |
| VPAT 2.5 EU | EN 301 549 only | EU customers only |
| VPAT 2.5 508 | Section 508 only | US federal only |
| VPAT 2.5 WCAG | WCAG 2.1 only | International, non-regulatory |

**Recommendation:** Use VPAT 2.5 INT for maximum coverage.

Download official templates: https://www.itic.org/policy/accessibility/vpat

## Conformance Terms (Critical to Use Correctly)

| Term | Meaning | When to Use |
|---|---|---|
| **Supports** | Product fully meets this criterion in all tested contexts | Use only when testing confirms full compliance |
| **Partially Supports** | Product meets criterion in some cases but not all | When some features/contexts comply, others don't |
| **Does Not Support** | Product does not meet this criterion | When testing confirms clear failure |
| **Not Applicable** | This criterion cannot apply to this product type | When the feature type simply doesn't exist |
| **Not Evaluated** | Criterion has not been tested in this evaluation | Acceptable to disclose — better than guessing |

**Common Mistake:** Marking "Supports" without testing, or marking "Not Applicable" to avoid evaluating something.

## VPAT Structure

### Header Section
```
Name of Product/Version: [Product Name v2.3]
Report Date: [Month Year]
Report Version: [1.0]
Product Description: [Brief description of what the product does]
Contact Information: [accessibility@company.com]
Notes: [Testing methodology, scope, tools used]
Evaluation Methods Used: [Automated: axe-core v4; Manual: NVDA+Firefox, VoiceOver+Safari; 
                           Keyboard-only testing; Zoom testing 100-400%]
Applicable Standards: WCAG 2.1 Level AA, Revised Section 508
```

### Criteria Tables
For each success criterion, provide:
1. **Criteria** — The criterion number and name
2. **Conformance Level** — Supports / Partially Supports / Does Not Support / Not Applicable / Not Evaluated
3. **Remarks and Explanations** — Specific details about what was tested, what works, what doesn't

---

## Writing Good Remarks

### Supports — Good Example:
> "All images throughout the product include meaningful alternative text. Decorative images use empty alt attributes. Complex charts include extended descriptions via aria-describedby. Tested with NVDA+Firefox and VoiceOver+Safari."

### Partially Supports — Good Example:
> "The majority of form fields include programmatically associated labels. The date range picker component uses custom styling without ARIA labels on the individual date inputs. All other form inputs fully support this criterion. Issue tracked for remediation in v2.4."

### Does Not Support — Good Example:
> "Video content in the media library section does not include synchronized captions. Automated captions generated by the platform are not synchronized accurately. This affects approximately 40 videos. Remediation planned for Q2 2025."

### Not Applicable — Good Example:
> "This product does not include any audio-only content. All video content includes both audio and video tracks."

### What to Avoid:
- ❌ "Supports" with no remarks
- ❌ "Not Applicable" for criteria that clearly apply
- ❌ Generic remarks copied from other VPATs
- ❌ Referring to future planned features as current compliance

## Section 508 Chapters (for US Federal)

Chapter 3: Functional Performance Criteria
Chapter 4: Hardware
Chapter 5: Software
Chapter 6: Support Documentation and Services

Most web applications primarily fall under:
- **Chapter 5 (Software)** — applies WCAG 2.0 Level AA for web-based software
- **Section 602** — Support documentation accessibility
- **Section 603** — Support services accessibility

## EN 301 549 for EU Compliance (EAA 2025)

EN 301 549 v3.2.1 is the harmonized European standard. It references WCAG 2.1 for web content and adds additional requirements for non-web software, documentation, and services.

The **European Accessibility Act (EAA)** requires compliance by:
- **June 28, 2025** — New products/services
- **June 28, 2030** — Existing long-term contracts

EAA covers: e-commerce, banking services, audiovisual media services, e-books, transport services, and more.

## VPAT Maintenance

VPATs should be updated:
- When a major new version of the product is released
- When known accessibility issues are remediated
- When new features are added that change the accessibility profile
- At minimum: annually

Include in the VPAT:
- Date of last evaluation
- Product version evaluated
- Testing methodology and tools
- Known issues and planned remediation timeline

## Frequently Asked Questions

**Q: Can I use an automated scanner to generate the VPAT?**  
A: No. Automated scanners only catch ~30% of WCAG issues. A VPAT must reflect manual testing. Using automated results alone creates false "Supports" ratings and legal liability.

**Q: What if our product has some accessibility issues?**  
A: Be honest. "Partially Supports" and "Does Not Support" with clear explanations and remediation timelines are far better than false "Supports" claims. Buyers respect transparency.

**Q: Do we need a new VPAT for each product version?**  
A: Yes, if the version has significant changes. Minor patch versions may not require a full new VPAT, but document the version clearly.

**Q: Who should complete the VPAT?**  
A: Ideally an accessibility specialist with hands-on testing experience, not marketing or legal without testing background. External specialists (like ADACP) provide maximum credibility.

**Q: What level of WCAG should we target?**  
A: WCAG 2.1 Level AA minimum (required by ADA, Section 508, EAA). WCAG 2.2 Level AA is now recommended as it supersedes 2.1 and is increasingly required.
